using on-exception in global transitions
Perhaps I'm not approaching this correctly, but I was hoping to be able to use the on-exception attribute in a global transition to catch the 'NoSuchConversationException' and restart the flow. Along the lines of:
Code:
  <global-transitions>
        <transition to="setupAction" on-exception="org.springframework.webflow.execution.repository.NoSuchConversationException"/>
    </global-transitions>
I'm finding in practice this doesn't work. The exception still bubbles up and I get the familiar "No conversation could be found with id ..." message.

We're being fairly aggressive in closing exiting a flow anytime the user leaves the page (we have a simple search in every page and do not want a number of abandoned flows floating around as a result) so we're finding that the back button results in this error. Our preferred behavior, as I mentioned, is to simply restart the workflow within a new conversation. Is there another way to reach the same result?

Rob,

NoSuchConversationException is a repository-level exception noting that access to the flow was disallowed (because there was no flow execution associated with the logical conversation with the submitted id). It is a higher-level exception--the flow execution itself never gets a chance to handle it because it does not exist.

With that said, we should support conversation / flow execution history--being able to to use the conversationId (or flow execution key) to determine what the flow definition was used for that historic task execution. Once the definition is known, it is trivial to start a new execution (restart)--just redirect to it.

This is currently not supported though, but will probably be in the 1.0.x timeframe. I believe there is a specific JIRA on history; there certainly should be.

The best offered out-of-the-box is to catch the exception at the SWF client level--e.g a Spring MVC HandlerExceptionResolver. Unfortuately there is no way to determine given the conversationId the flow definition it pertains to--not yet.

I guess I was thinking something along the lines of redirecting to the flow instead of throwing the exception, but this works, too.

Code:
Code:
public class NoSuchConversationExceptionResolver extends
        SimpleMappingExceptionResolver {

    public NoSuchConversationExceptionResolver() {
        // otherwise the exception info will appear in the URL
        setExceptionAttribute(null);
    }

    @Override
    public ModelAndView resolveException(HttpServletRequest request,
            HttpServletResponse response, Object handler, Exception ex) {
        ModelAndView modelAndView;

        if (ex instanceof NoSuchConversationException) {
            modelAndView = getModelAndView("redirect:"
                    + request.getRequestURI(), ex, request);
        } else
            modelAndView = super.resolveException(request, response, handler,
                    ex);

        return modelAndView;
    }

}
Bean definition:
Code:
    <bean id="noSuchConversationExceptionResolver"
        class="org.freeshell.rkm.NoSuchConversationExceptionResolver">
        <property name="mappedHandlers">
            <set>
                <ref bean="flowController"/>
            </set>
        </property>
    </bean>
I should note that I'm using the RequestPathFlowExecutorArgumentExtractor and so can ignore the query parameters.

Rob,

The rationale there is the concept of a flow execution should not be coupled with that of a conversation. We've still yet to announce all this, but as you'll see with 1.0 RC2 the repository interface deals strictly with flow executions--there is on longer a "first class" concept of a conversation within SWF. Behind the scenes, most of the repository implementations delegate to generic conversation service to track conversational and continuation state. This means several things:
- you can have a repository impl without the 'conversation' concept
- there are less concepts in swf core---flow and flow execution, that's it.
- much more is now abstracted behind the flow execution repository subsystem, including the concrete structure of the flow execution key, and any delegation to other systems such as a conversation manager.
- the conversation service is a portable abstraction that can be reused outside of SWF.

Getting back to the exception, it wouldn't make since to have a "NoSuchConversationException" now as part of the root repository exception hierarchy. "NoSuchFlowExecution" is a fit, since it makes sense at level. What other types of failures are you interested in catching? We're considering some sort of "access to flow execution denied" exception--right now we simply throw NoSuchFlowExecution for most repository failure scenarios (NoSuchConversation, ContinuationNotFound, InvalidContinuationId). Those concrete causes only make sense to the specific repository impls--but we want to make sure we have the "logical" exception types covered to support the handling needs of clients...

The idea is to have an exception hierarchy that makes sense for all repoistory implemetnations and is relavent to clients for handing/recovery.

Hi Keith,

I think you should consider an exception hierarchy somewhat analagous to that used with Spring's DAO support. At the very least, I would make certain that there are 2nd-level exceptions (i.e., subclasses of the 1st-level FlowException) that cover the following topics:
  • undefined or non-existent: flow / flow definition, (NoSuchFlowException).
  • invalid: invalid execution state, invalid parameters, etc. The existing StateException hierarchy could extend this, or this could be an additional branch covering invalid operations not related to flow states.
  • inaccessible: "access denied", etc. could be subclasses
  • resource failure: e.g., for filesystem or database backed repositories, etc.
  • uncategorized

Exceptions specific to continuations would naturally be subclasses of the appropriate 2nd-level exceptions outlined above.

One thing that puzzles me currently (at least in the 1.0 EA code base) is that NoSuchStateException does not lie under the StateException hierarchy. This seems illogical to me.

What do you guys think of such a broad 2nd-level hierarchy?
