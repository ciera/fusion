Why are you putting the object in the session yourself? The controller can and should do that for you. The formBackingObject should only be there for creating the initial formBackingObject. With the sessionForm property you can control wether it is stored in the session or not.

Further more for checking fields empty, invalid values write a Validator which you register with your Controller. If there are errors the onSubmit method will not be called but instead the screen will be redisplayed with the errors.

I suggest you read-up on SpringMVC, read chapter 13 of the reference guide. Also checkout the samples shipped with Spring, they make all the above clear. Further more use the search on the forum, this question has been answered a couple of times.

The controllers are by default not putting anything in the session. In your configuration set the sessionForm property to true for that to work. So you don't have to check anything yourself.

rewrite your formBackingObject to this, that is all you need. The Controller will take care of the rest.

Code:
protected Object formBackingObject(HttpServletRequest request) throws Exception {
  return concreteCabeceraSalidaAlmacenImpl.getCabeceraSalidaAlmacenIdGeneradoBO();			        		
}
In your configuration

Code:
<bean id="yourController" class="ControllerClass">
  <property name="sessionForm" value="true"/>
  <property name="commandName" value="CabeceraSalidaAlmacenCommand"/>
  <property name="formView" value="transaccionSalidaAlmacen"/>
  <property name="succesView" value="inicio" />
</bean>
Also I checked your controller a little closer but what are you trying to achieve in your referenceData method?

The refresh is being processed as a new call to the controller. All GET request are interpreted as a fresh/new request and thus a commandObject is getting created.

When POSTing a form the form is going to be retrieved from the session.

If you want to change this behavior override the isFormSubmission method to include your own logic.
