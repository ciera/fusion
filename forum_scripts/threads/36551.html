The problem you have is that the <action bean="mamaloe" method="getFullList"/> will always look for the "mamaloe" bean in the application context.
Instead of a normal action, use an evaluate action like so:

<evaluate-action expression="flowScope.mamaloe.getFullList()">

Check the "numberguess" sample application for more examples of the evaluate-action.

The 'missing link' here is the flow variable. If I understand correctly you want to map a bean defined in a Spring application context to a subflow. Such a bean is not directly available in flowScope (or any of the other scopes). So before you can map it you'll have to make it available in flowScope. You can do that with a flow variable like so:

<var name="formBAction" bean="formBAction" scope="flow"/>

You can't pass the RequestContext to a method using the <evaluate-action/>. The reason is that by using the <evaluate-action/> (or <bean-action/> for that matter), you're trying to invoke methods on objects that are not SWF dependent. Passing the RequestContext would ofcourse introduce a SWF dependency in the called code, so in that case you should just use a direct <action/> (i.e. subclass MultiAction) and implement your method there.

You can use an <action/>! You can't directly lookup the action to use in the application context, but you could implement a generic action that delegates to an action (or POJO bean) in flow scope. Something like this:

Code:
public MyAction implements Action {
  public Event execute(RequestContext context) throws Exception {
    Action actionFromFlowScope = (Action)context.getFlowScope().get("myAction");
    return actionFromFlowScope.execute(context);
  }
}
Your subflow then just has a direct reference to this 'delegating action' and not to the actual action that is used.
