Hi tboonyar,

Are you using an event ID of "back" on your step 2 and step 3 back buttons ?

If not, make sure you are setting the "backEventId" property of the listener:

XML Code:
HTML Code:
    <bean id="flowCancelListener" class="com.mycompany.BackToLastViewStateFlowExecutionListener">
        <property name="backEventId"><value>myBackButtonEvent</value></property>
    </bean>

    <global-transitions>
        <transition on="myBackButtonEvent" to="${flowScope.previousViewStateId}"/>
    </global-transitions>
JSP Code:
HTML Code:
<a href='myflow?_flowExecutionKey=<c:out value="${flowExecutionKey}"/>&_eventId=<c:out value="${myBackButtonEvent}"/>' target="_blank">
    Go Back
</a>
The reason for this is that the listener checks the last event ID, and if it matches the configured event ID, it will remove the last view state (because the last view state would send you forward, not back). Otherwise, it simply adds the last view state to the stack, which would explain the behaviour you are describing.

Java code:
PHP Code:
        // check the last event ID against the configured event ID
        if (context.getLastEvent().getId().equals(backEventId)) {
            // If this was a back event, then remove the last view state, as this
            //     will now represent a "forward" rather than a "back"
            viewStates.removeLast();
            previousStateId = viewStates.isEmpty() ? "" : viewStates.getLast();
        } else {
            // this wasn't a back event, so add the last view state to the stack.
            previousStateId = previousState.getId();
            viewStates.add(previousStateId);
        }

        context.getFlowScope().put("previousViewStateId", previousStateId);
