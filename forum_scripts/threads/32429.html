When you create an object with new it isn't a Spring managed bean and hence not being injected with anything or under Spring management. Assuming you already running some kind of application you already have a applicationContext.xml (or whatever the name is you specified). For more information check the first few chapters of the Spring reference guide.

Configure your bean as a prototype and retrieve instances from the applicationcontext.

Code:
<bean id="myObject" class="MyObject" scope="prototype"/>
Then from some other spring managed bean

Code:
MyObject object = (MyObject) context.getBean("myObject");

Quote:
But since getApplicationContext() always returns null, something is not right.
Wel actually it is

First of all if you want to have the applicationContext injected it MUST be a spring managed bean. If it isn't your ApplicationContext isn't going to be injected. So object created with new SomeObject implementing ApplicationContextAware are never going to be injected with the applicationcontext.

Next your applicationContext is present AFTER construction/initialization of the object. Hence your null in the constructor. For more information about the Spring Lifecycle check chapter 3 of the reference documentation.

Change your getSomeBean method to something like this (and remove the private ApplicationContext ctx line, you can access it by getApplicationContext, no need to keep 2 references to the same object...).

Code:
public SomeBean getSomeBean() {
    return (SomeBean) getApplicationContext().getBean("someBean");
}
If you override the isContextRequired method in your class to return true, you will receive an error when calling getApplicationContext and if it is null.

Quote:
As I wrote, MyObject extends ApplicationObjectSupport. That should enable MyObject to access the ApplicationContext using the getApplicationContext() method.
I want this because then I can fetch beans, using applicationContext.getbean("someBean"), from MyObject.
Only if MyObject is also managed by Spring. Simple doing a new MyObject() from some bean/class managed by Spring doesn't inject the ApplicationContext. It just creates an object outside the Spring container and treats it as a normal object. As if you where just doing new Long(12); It is just a plain object nothing managed by Spring.

Finally the preferred way to get a reference to a bean is to inject it into a bean, not pulling it from the context. You could even define your bean as a prototype and have all of it's dependencies injected when you get a new instance from the ApplicationContext of create a FactoryBean which creates those objects for you.

Quote:
But since getApplicationContext() always returns null, something is not right.
Don't just new objects and make sure you understand the Spring lifecycle.

Code:
public class SpringBeanManager extends ApplicationObjectSupport {

    /** Name of application context bean. */
    private static final String SOME_BEAN = "someBean";
    
    /** Name of application context bean. */
    private static final String ANOTHER_BEAN = "anotherBean";

    /** Reference to application context inside the singleton. */
    private static ApplicationContext applicationContext;

    /** Static inner class that holds a SpringBeanManager object. */
    private static class SpringBeanManagerHolder {
        private static SpringBeanManager instance = new SpringBeanManager();
    }

    /** Default construtor. This class should not be instantiated externally. */
    private SpringBeanManager() {
        super();
    }

    /**
     * Returning the singleton instance.
     * 
     * @return The instance.
     */
    public static SpringBeanManager getInstance() {
        return SpringBeanManagerHolder.instance;
    }

    /**
     * {@inheritDoc}
     */
    @Override
    protected void initApplicationContext() throws BeansException {

I think it is better to wire everything up in the applicationContext.xml, this way your code doesn't clutter with all the SpringBeanManager.getInstance().getSomeBean(); calls. You can just use the service/dao/... which you injected. Which is why the Inversion of Control libraries (like Spring) exist.

What you are trying to do is to use the IOC container but still not use it to the full extend, wrapping it in an (evil) singleton and still doing lookups from your code.

Currently we have some 30 services, 40 dao's, 15 validators, hibernate, datasources etc. wired up by Spring, no more lookups in our code, our code doesn't even care if the datasource comes from a jndi lookup, or even if it is a local datasource. It is all transparent due the the IOC container. I should have known Spring years before, it makes things so much easier .
