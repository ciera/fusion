I did some additional debugging and it looks like the Errors object gets overlayed with a new one created during form population after validation. This is in the setupForm and bind methods of FormAction - depending on which one I used, I saw the same issue.

Now changing this:

Code:

	doBind(context, binder);
	setFormErrors(context, binder.getErrors());
	return binder.getErrors().hasErrors() ? error() : success();
to this:

Code:

	doBind(context, binder);
	Errors errors = binder.getErrors();
	errors.addAllErrors(getFormObjectAccessor(context).getFormErrors(getFormErrorsScope()));
	setFormErrors(context, errors);
	return binder.getErrors().hasErrors() ? error() : success();
Seems to carry over the validation errors and fix the problem I was seeing. Not sure if it has some other side effects or if there is something else wrong that causes something to be called out of order.

On postback, when bindAndValidate is called, it returns an "error" which causes the transition not to be executed and your view state to re-enter itself. This reentering process triggers the entry-actions (and thus a setupForm invocation), which because bindOnSetupForm is set to true, wipes out the existing errors instance...

Well, one obvious solution there is to break out the entry-action logic into its own action state, and enter that state before your view state (e.g. having a dedicated 'setupForm' action state).

But the bigger question is why you have to even set bindOnSetupForm to true in the first place. Are you using the latest code? I'm looking at the source now and it looks like createBinder (and thus initBinder) is still called on setupForm even if bindOnSetupForm is set to false. That would install your property editors before the form is displayed. Take a look!

I'm a bit confused. What was going on exactly and how did you resolve this issue? Perhaps if you could post your flow definition...

Some thoughts:

If you have a form object that is updated over the life of the flow, make the form object's scope FLOW. That form object will remain in scope for the life of the flow automatically.

The data binder/Errors behavior is completely separate beast. By default, when setupForm is called an Errors instance is created, property editors are installed, and the Errors instance is placed in REQUEST scope (and thus automatically exposed to the view when the next view-state is entered in that request).

This means Errors only exist on a per request basis by default. So if you call setupForm, then enter a view state, then call bindAndValidate on submit, a entirely new Errors object will be created when data binding and validation occurs on the next request (again with property editors installed), which is generally what you want.

Likewise, if you go from bindAndValidate to the next view state in the form, the Errors instance will still be in scope because it's still the same request (and presumably empty since validation succeeded), and will again be automatically be exposed to the view, again with all property editors already installed. Which is again generally what you want.

It shouldn't be any more complicated than that. Just call setupForm before your first view state, call bindAndValidate on your submit transitions, use the validatorMethod property to drive piecemeal form validation, and make sure the form object is in flow scope. Sellitem demonstrates exactly this.

Thomas,

I think I know what is going on, but I really am having a long hard time debugging because you're not including the whole picture.

First off, it seems you are using a separate form action for each page of your form. My first question would be why on earth are you doing that. One form action is designed to bind edits across all forms of a flow so long as those forms update properties on the same form object. You typically only need multiple form actions if you have multiple, independent form objects--I would NOT expect more than one form action per flow for most cases.

So if you are indeed doing this (I can't tell because you didn't include the critical form2Action definition OR important form2 view-state definition for some reason--I'm only guessing b/c of the "hints" you keep leaving me), it's no wander these dang editors are not being installed. In this case I'm sure what's happening is form1Action is not installing any editors, and then when form2Action runs it doesn't install it's editors because it sees an errors instance already in scope for the same form object put there by formAction1...

I would recommend using one form action, have it install ALL THE EDITORS needed accross all form pages--and use a propertyEditorRegistrar, don't override initBinder. You'll simplify your number of form actions from one per page to one total and you'll centralize your property editor registration logic into a tiny PropertyEditorRegistrar class.

Finally, there is no need to call setupForm multiple times for form2, form3, etc. Just call it once when the flow starts before displaying form1.

OK
For anyone else that runs into this issue...
If you have bindOnSetupForm enabled. DO NOT put your setupForm action in as an entry-action on your view-state.

In other words, do this:

As an aside, this thread is a bit out-of-date since it discusses pre-1.0-final versions of SWF except for the last 2 posts.

Quote:
I am still having to do "<action bean="myFormAction" method="setupForm"/>" in my render-actions when using redirect-on-pause to re-register my custom property editors in a PropertyEditorRegistrar. This seems like a waste of processor time. Is this not the case?
The problem is that the PropertyEditors are transient in the Errors object, so they get lost inbetween requests since the Errors object is serialized out by the flow execution repo. This is the reason we need to re-register the PropertyEditors every time we want to render a view.

Quote:
Is there any reason why this re-registering isn't done automatically for rendering, but *is* done for bindAndValidate?
Well, the "setupForm" action method is re-registering the property editors and the "bindAndValidate" action method does the same thing. It's not really a question of doing it 'automatically': you have to call some action method on the FormAction that does the re-registration.
