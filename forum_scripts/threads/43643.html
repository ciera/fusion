Thats right.

Anything in flash and flow must be serializable.

If you using JSF you have 2 options.

1. Use the MultiAction/FormAction pattern which means you have this stateless class supporting your flow definition (like in the booking-jsf example). This is good because you can safely inject your pojo services there, but but bad because in JSF you really want these objects in your backing beans.

2. Put your pojo services in your pojo backing beans (no multi action or form action at all required). You then declare your backing beans in the flow def. I like to think of this as a class but in xml - so declare your backing beans for the flow using var tags. Make sure if you want pojo stateless services in the BB, the BB must be 'conversation' scoped.

Code:
<var name="exampleFlowBackingBean" scope="conversation"/>
Next, dont access your beans like in the booking-jsf example, the reason for this is a) because you dont need to and b) because you dont have a handle on RequestContext using this method.

Declaring var creates the bean and puts it in scope, so to call operations on it use evaluate-action.

Code:
<evaluate-action expression="conversationScope.exampleFlowBackingBean.setXXX()"/>
If you use this method to set an associated variable:

Code:
<bean-action bean="exampleFlowBackingBean" method="setYYY()">
...you will NOT be acting on the same 'exampleFlowBackingBean' (but if that was evaluate-action you would be). This is why in the booking example the code always fetches the state from the RequestContext - but I feel this is unnecessary.

Try both options and see which you prefer.

Marcelo,

The reason that beans you scope to the Web Flow custom scopes must be serializable is that is how the continuation snapshots are stored.

Since the exception you are getting is from a JSF component, I would imagine you are either using the binding attribute to inject that component reference into the bean or else creating the component from scratch in your bean and inserting it into the tree. Either way, the component does indeed need to be marked transient.

I would not say that you are using the scope incorrectly. If you are in fact using the binding attribute, you should safely be able to mark the component as transient and then the proper instance should be re-injected into the bean by JSF during the RESTORE_VIEW phase of the next request.

Hi James,

Actually, I was speaking with regard to SWF 2.0-m1 and the use of the custom Spring 2.0 scopes, since that's what Marcelo seems to be using. I certainly understand your point that there are many different ways to approach things under 1.x. One of the motivations behind the custom scopes is to hopefully simplify the programming model a bit, to the point where using the custom scopes should result in very close to the same behavior that a JSF developer would expect if they could specify flow scope in their faces-config. We're really trying to get to a model where the ideal in all cases is for all of the beans to be managed by Spring, while still providing good support to existing JSF users who are perhaps new to SWF and already have invested significant time into specifying managed beans in their faces-config. The approach I outlined should ultimately work regardless of whether or not the scoped bean implements any SWF specific interfaces.
