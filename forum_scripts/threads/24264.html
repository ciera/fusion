Given subflow outputs are explicit, should none-existant outputs be exceptional?
Keith (all),

Now that a subflow must explicitly expose attributes that can be copied at the end of it's lifecycle, should requests for output-attributes be exceptional, i.e. throw an exception?

At the moment they are simply set to null, but this seems a bit flaky and not very helpful.

To put it another way, isn't it a misconfiguration for a flow to expect an attribute which the subflow isn't explicitly exposing? When is this ever valid?

To be clear:

Code:
(subflow)
<flow ...>
  <end-state id="filesCreated">
    <output-attribute name="emptyFiles"/>
    <output-attribute name="createdFiles"/>
  </end-state>
</flow>

(parent flow)
<flow ...>
  <subflow-state id="createSingle.action" flow="mySubflow">
    <attribute-mapper>
      <output-mapping name="emptyFiles"/>
      <output-mapping name="createdFiles"/> <!-- this isn't exposed!!!-->
      <output-mapping name="duplicateFiles"/>
    </attribute-mapper>
    <transition ...  />
  </subflow-state>
</flow>
I think is an invalid configuration and should throw an exception...

Just a thought

Hi Colin, Keith, et al,

Quote:
Originally Posted by yatesco
Am I the only one who thinks this is a good idea?
Well, if this functionality were to be added, I think it should be provided via an additional, configurable option. In other words, I don't think this should necessarily be the default behavior. I can imagine situations in which you either a) don't care if the parameter is mapped, or you b) handle null values explicitly elsewhere.

In summary, I'm not so sure that having to handle such an exception would be welcome and/or useful in all situations. It adds (perhaps unnecessary) dependencies and potentially inhibits backwards compability with existing components/flow definitions.

@Colin: to answer your original question, considering it a misconfiguration for a flow to expect an attribute which the subflow isn't explicitly exposing is certainly a reasonable assumption some of the time, but not all of the time. If you were to enforce exception handling for all use cases by default, you would essentially be binding the hands of the developers. It's somewhat analogous to throwing Exceptions vs. RuntimeExceptions in a framework API.

When is this ever valid? -- Let the developers decide that for themselves.

my 2 cents,

Quote:

But you *do* care because you are explicitly defining an output-mapping in the subflow-state. An output-mapping that can never be resolved.

Quote:
Originally Posted by sbrannen
b) handle null values explicitly elsewhere.
I think the issue of a null value is misleading I am not suggesting that a null value for an attribute that is exported by a subflow is an exception state.

For me it is analagous to the spring container ignoring incorrect bean definitions....or more technically the XML parser ignoring definitions that do not fall within the DTD.

Quote:
Originally Posted by sbrannen
@Colin: to answer your original question, considering it a misconfiguration for a flow to expect an attribute which the subflow isn't explicitly exposing is certainly a reasonable assumption some of the time, but not all of the time. If you were to enforce exception handling for all use cases by default, you would essentially be binding the hands of the developers. It's somewhat analogous to throwing Exceptions vs. RuntimeExceptions in a framework API.
I disagree The discussion re: runtime/checked exceptions is to do with whether an exceptional state is considered recoverable. This discussion is whether (what I term) a misuse of the subflow API (i.e. expecting attributes the subflow isn't exporting) is an exceptional state.

I really do think of it in terms of a misuse of the API exposed by the subflow, and that is *always* a misconfiguration.

To be doubly clear; I do not care whether the values exposed by the subflow is null, I do however consider asking the subflow for an attribute it hasn't exposed akin to an UnsupportedOperation/IllegalArgumentException.

Or, for another analogy; it is the same reason I think the compiler rejecting the following code a good thing:

Code:
  return true;
  a = b; // compiler falls over
To stretch to another point it is the same situation (for me) as referencing a bean in a flow that isn't exported in the bean factory.

Maybe a compromise would be for the builder to log a warning instead of throwing an exception.....

I think silently continuing would be the *wrong* thing because 99% of the time this situation will occur because of spelling mistakes, and it would be useful to know.
