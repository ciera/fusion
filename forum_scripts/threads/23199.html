Ok,

This is going to sound horrendously patronising but it's probably worth taking some time to read through the spring docs to understand how a form object (or command object as they call it) is used after submitting the HTML form.

What you've described can be characterised as follows: your list of products is "reference data" that is used in the JSP. If you have a look through the SimpleFormController class hierarchy you'll find a method named referenceData() - or something like that - that you can override to make the List available as a request attribute to your JSP.

Now consider what you're submitting - just one value, that the user selected in the drop-down. You need to specify a form object that can capture this. Spring MVC does a great job of "binding" the raw, String request parameters to the Form object's properties. So, say a product's ID is a Long, I would create the following class and set it as the controller's commandClass:
Code:
public class MyFormObject {
    private Long productId;
    public MyFormObject(){
    }
    public void setProductId(Long productId){
        this.productId = productId;
    }
    public Long getProductId(){
        return productId;
    }
}
Spring will now automatically convert the parameter "productId" to a Long and set it on a form object it instantiates, and pass it into the onSubmit(...) method.

The reason for the getter is that if an error occurs during the binding, for example if it attempts to convert "aabb" to a Long, the bad value can be displayed in the formView again for the user's attention, and the bind tag handles this.

They allow you to specify how you want different properties from your form object to be displayed. You can add your own PropertyEditors so that Spring can automatically convert "2006-03-14" into a valid Date object for you, etc.

HTH. Now read the docs...

My firm belief is that for any wizard-type page flow the most intuitive way of chaining them together is to use Spring Webflow. This allows you to describe your page flow in an XML file and hand off the action parts of the process to a bean. It has slightly more flexible validation than Spring MVC as well (AFAIK).

Consider this:
Code:
<flow start-state="prepareForm">

    <action-state id="prepareForm">
        <action bean="deleteProductAction" method="loadReferenceData"/>
        <transition on="success" to="deleteProduct.view"/>
    </action-state>
    
    <view-state id="deleteProduct.view" view="deleteProduct.logicalJspName">
        <entry-actions>
            <action bean="deleteProductAction" method="setupForm"/>
        </entry-actions>
        <transition on="submit" to="confirmDeleteProduct.view">
            <action bean="deleteProductAction" method="bindAndValidate">
                <attribute name="validatorMethod" value="validateDelete"/>
            </action>
        </transition>
    </view-state>
    
    <view-state id="confirmDeleteProduct" view="confirmDelete.logicalJspName">
        <transition on="submit" to="deleteProduct.action"/>
        <transition on="cancel" to="deleteProduct.view"/>
    </view-state>
    
    <action-state id="deleteProduct.action">
        <action bean="deleteProductAction" method="deleteProduct"/>
        <transition on="success" to="deleteProductSuccess.end"/>
        <transition on="error" to="deleteProductError.end"/>
    </action-state>
    
    <end-state id="deleteProductError.end" view="deleteProductError.logicalJspName"/>
    
    <end-state id="deleteProductSuccess.end" view="deleteProductSuccess.logicalJspName"/>
    
</flow>
Try to match the to="stateName" codes with the id="stateName" to see how the flow hangs together. Webflow rocks...
