To validate correctly implement the validatePage method (see example)

Code:
protected void validatePage(Object command, Errors errors, int page) {
  switch(page) {
    case 0;
      validatePageOne(command, errors);
     case 1;
     validatePageTwo(command, errors);
  }

}

private validatePageOne(Object command, Errors errors) {
  Collection<Validator> validators = getValidators(Quotation.class);
  for (Validator validator : validators) {
    validator.validate(command, errors);
  }
}

protected Collection getValidators(Class clazz) {
  List validators = new ArrayList();
  for (Validator validator : getValidators()) {
    if ( validator.supports(clazz) ) {
      validators.add(validator);
    }
  }
  return validators;
}
A sample validator

Code:
public class QuotationValidator implements Validator {

      errors.rejectValue("quotation.branche", "required");
    }
    
  
  }
}

and y u recommend command.quotation as my command object name is quotation
Your commandName is command which you can deduct from this error message.

Code:
Field error in object 'command' on field 'insurerObject.insurerTypeId'
The fieldname MUST be the fullname path of your property. Your current fieldname will result in the following java call.

Code:
command.getInsurerObject().getInsurerTypeId()
Now I'm quite certain that you don't have a getInsurerObject() on your commandobject, because (as I deduced from yuor jsp where you use the nested paths) it is a property of the quotataion object

Code:
<spring:nestedPath path="quotataion">
<spring:nestedPath path="insurerObject">
So what you actually want is a call to command.getQuotataion().getInsurerObject().getInsu rerTypeId(). Which can be accomplished by using the correct and thus full fieldname/field path of your property. The full path is always deduced from the command object. So in your and this specific case it should be the following

Code:
ValidationUtils.rejectIfEmpty(error,"quotataion.insurerObject.insurerCode","Insurer Code Cant be Empty");

Your validator and fieldnames look ok.

Your controller has some issues. Never keep state in your controller Your controller is a singleton and subsequent request may override these properties. Especially your model, when 1 request is processing and another comes by it does a new HashMap and gone is your previous model

Code:
public class QuotationCntlr extends AbstractWizardFormController {
   //Don't keep state!!!
    ModelAndView modelAndView;
    Map<String,Object> model;
    private QuotationDAOImpl dao;

    public QuotationCntlr() {
    }
Next in your referenceData method you set your formbackingobject in the model?! Why? Also you are setting stuff directly on the request why? Use the model for that and let Spring handle the 'hard' stuff for you.

Code:
    protected Map referenceData(HttpServletRequest request, Object command, Errors errors, int page) throws Exception {
        //Each request needs a new Map, so recreate it.
        Map model = new HashMap<String,Object>();
        if(page == 0){
            model.put("branches",dao.getBranches());
        }
        else if(page == 1){
            model.put("vh",dao.getVehiclesDescription());
        }
        else if(page == 2){
            model.put("policyCoverTypes",dao.getPolicyCoverTypes());
            model.put("companies",dao.getInsuranceCompanies());
        }
        return model; 
    }
The getter for the dao, isn't needed, you only need the setter (best maintainable code is code not written ).

Your jsp it looks quite messy.

Code:
<td width="11%"><spring:nestedPath path="quotation"></td>
Does your QuotationImpl have a property quotation? I don't see it referenced anywhere in your code (and you didn't include the code for InsurerImpl). So I think you should/must remove this nested path (on a sidenote why aren't you using the new form tags make your jsp much mor readable).

In the top of your jsp you have

Code:
<spring:hasBindErrors name="command">
  <c:forEach items="${errors.fieldErrors}" var="err">
    <li><spring:message message="${err}"/></li>
  </c:forEach>
</spring:hasBindErrors>
I suggest you try

Code:
<spring:hasBindErrors name="*">
  <c:forEach items="${errors.fieldErrors}" var="err">
    <li><spring:message message="${err}"/></li>
  </c:forEach>
</spring:hasBindErrors>
[/code]

Let spring work for you, you are trying to achieve stuff which spring provides for you. You are working against the framework. I (again) strongly suggest you take a look at the Petclinic and Petstore sample applications shipped with spring. They make a lot clear in what to use and what not to.

I think you are struggling with and don't quite yet understand the basics.

I was talking about the new spring 2.0 form tags. Next to this I suggest you read chapter 13 of the reference guide, especially chapter 13.3 which explains about Controllers.

Again look at the samples (PetClinic/PetStore) which explain a lot about how to use Spring MVC.

Some pointers, all the binding should be done on the command object (which is default named command). This object will/must be created in the formBackingObject method. Your command object is a container, it holds a reference to the 3 objects you are managing. So it should only have 3 get/set pairs. The rest is on the enclosed objects.

The data which is being put in the Map in the referenceData method is available in your jsp under the same name as you put in the Map. So no need to add it to the request yourself.
