No it hasn't... The createFormObject method IS only called when no formObject does YET exist. Which is only at the start at the flow, after that the object will be stored in the flowScope.

After a redirect you loose all properties stored in the requestAttributes (which in the end is where all properties from all flow scopes end up).

Read the tip again and try to really understand what is going on behind the covers. Check the source code and don't assume things.

Code:
public Event setupForm(RequestContext context) throws Exception {
	if (logger.isDebugEnabled()) {
		logger.debug("Executing setupForm");
	}
	// retrieve the form object, creating it if necessary
	Object formObject = getFormObject(context);
	ensureFormErrorsExposed(context, formObject);
	return success();
}

protected Object getFormObject(RequestContext context) throws Exception {
    FormObjectAccessor accessor = getFormObjectAccessor(context);
    Object formObject = accessor.getFormObject(getFormObjectName(), getFormObjectScope());
    if (formObject == null) {
        formObject = initFormObject(context);
    }
    else {
        if (logger.isDebugEnabled()) {
            logger.debug("Found existing form object with name '" + getFormObjectName() + "' of type ["
                    + formObject.getClass() + "] in scope " + getFormObjectScope());
        }
        accessor.setCurrentFormObject(formObject, getFormObjectScope());
    }
    return formObject;
}

private Object initFormObject(RequestContext context) throws Exception {
	if (logger.isDebugEnabled()) {
		logger.debug("Creating new form object with name '" + getFormObjectName() + "'");
	}
	Object formObject = createFormObject(context);
	putFormObject(context, formObject);
	return formObject;
}
In setupForm the getFormObject method is called. If no formObject is currently found call the initFormObject method, the initFormObject method will delegate the call to the createFormObject method. When returning from createFormObject the formObject will be put in the flowScope (or the scope you configured).

If you read my first post on the other topic I mentioned, you can see I've gone through SWF code and found that createFormObject gets called , as you've said, only when no formObject does yet exist in form object scope. After working on a flow for some reason (my mistakes in code) I doubted that what I've seen in code, so I thought it wasn't working like that. Silly me. But I was also mislead by the FormAction JavaDoc.

Joshua Bloch in his well known presentation "How to Design Good API and Why it Matters" mentioned several characteristics of a good API, one of which is and I quote: "Easy to use, even without documentation". Regarding this issue, SWF not only isn't easy to use without documentation, but it's documentation for this part is misleading (in JavaDoc for FormAction it is suggested "You may override this (createFormObject) to customize where the backing form object instance comes from"). I wonder, why would anyone put such dependency in setupForm, and not let it's audience know about it at least through JavaDoc for createFormObject, just by stating that this method, regardless if it's overridden or not, will get called only if form object isn't found in form object scope (defined or default) so developer doesn't have to go into the framework code.
