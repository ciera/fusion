I have looked into this in a bit more detail. The call tree for ConversationContainer.createAndAddConversation is like this:

ConversationContainer.createAndAddConversation
SessionBindingConversationManager.beginConversatio n
AbstractConversationFlowExecutionRepository.genera teKey
FlowExecutorImpl.launch
FlowRequestHandler.handleFlowRequest
FlowController.handleRequestInternal

There is no locking or synchronization happening anywhere, except for synchronized methods in ConversationContainer. I studied the code and I don't immediately see how it could lead to a deadlock: the logging statement you see was logged *inside* createAndAddConversation in SWF 1.0.0, so the active thread must have acquired the monitor on the ConversationContainer object by that point since createAndAddConversation is synchronized. The 'conversations' list is not synchronized, so the "conversations.add(conversation);" should not block. In case the max number of conversations is exceeded, ContainedConversation.end (unsynchronized) will be called, delegating back into ConversationContainer.removeConversation. That method is also synchronized but that should not be a problem since the active thread already has the monitor!

Getting that thread dump would be very valuable!

What version of SWF are you using? Seems like 1.0.0. Try to upgrade to 1.0.3 and see if you can still reproduce the problem.

Since you're using linux it should be very easy to generate a thread dump:

1) launch the application in a stand-alone tomcat
2) do whatever is necessary to get the problem
3) in your terminal, do 'ps -fu user' where user is the user that started tomcat, that should give you a list of processes, telling you the PID of the java process running tomcat
4) again in the terminal do 'kill -QUIT tomcatPID' with the PID of tomcat obtained in step 3
5) you should now have a thread dump in logs/catalina.out
