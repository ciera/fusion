In order to resolve beans from the ApplicationContext via JSF expressions like this one:
Code:
actionListener="#{uiLogin.submitButtonListener}"
You need to also have the following VariableResolver definition in your faces-config:
Code:
<variable-resolver>org.springframework.web.jsf.DelegatingVariableResolver</variable-resolver>
Also, defining the "uiLogin" bean with prototype scope as you have is not necessarily the best idea. It will probably not exhibit the behavior you are expecting. I would recommend scoping it to the request, either via the new Spring 2.0 scope, or by defining the bean in faces-config instead.

ActionListeners do not signal navigation by themselves. You have to actually signal some sort of outcome, either by hardcoding it in the action attribute of the commandButton, or binding to an Action method in the bean. In this case you would opt for the latter since navigation is conditional based on the result.

You could change your ActionListener method to an action, but I personally prefer to keep just one action method per bean, with multiple ActionListeners that can choose whether or not to set an outcome.

Following this logic, you would make the result a property of your bean instead of a local variable.

Something like the following:
Code:
public class MyBean
{
  private String result = "";
  
  public void submitButtonListener(ActionEvent e) {
    try {
      getLoginService().authorize(this);
      setResult(Constants.LOGIN_SUCCESS);
    } catch (AuthException authException) {
      setResult(Constants.LOGIN_FAILURE);
    }
  }

  public String signalEvent()
  {
    return getResult();
  }

  ...
}
Then your commandButton would look as follows:

Code:
<ice:commandButton type="submit" value="Submit"
			actionListener="#{uiLogin.submitButtonListener}" 
                        action="#{uiLogin.signalEvent}"/>
Of course, this is more of the traditional way of doing things in JSF, and you would just be letting SWF handle the view selection. But it is certainly possible to leverage the more declarative style of SWF even further by externalizing the call to the LoginService in a bean invoking action state. In that case you could eliminate both the ActionListener and Action methods entirely, just hardcoding an event in the commandButton's action attribute and letting SWF do the rest.
