getAsText not called when form loads
Hi:

I'm having a problem that I've seen references to in the forums and elsewhere, but I'm not finding any solutions. I'm using a SimpleFormController, and calling initBinder to register custom editors for handing various form properties. Everything works great on validation and submission, but getAsText is not being called at all when the form loads. This results in form fields not being pre-selected correctly. Here's an example:

I have a form in which a user can update their address. This includes a drop-down list of countries. The user has a country that they have selected previously. So I have a form backing object that contains a list of countries and a user who has one country. I'd like the full list of countries to show up on the form with the correct country pre-selected.

However, the countries in the list are (for whatever reason) not exactly equivalently to the country the user has. I'm expecting the CountryEditor.getAsText method, which returns a country's ID as a string, to be called so the IDs of the countries can be compared, rather than the entire country object. But getAsText is never called. The countries are compared as objects (using equals()).

I see one thread about this issue, with no resolution:
http://forum.springframework.org/showthread.php?t=32196

This thread references SPR-2915:
http://opensource.atlassian.com/proj...rowse/SPR-2915

However, SPR-2915 indicates the issue as fixed, more or less "unable to duplicate."

So I guess my question is two fold:

Does anyone have a solution to this issue?

If not, how can I get SPR-2915 open again? I am happy to provide everything that is needed to duplicate the problem. I'm just not sure how to go about doing that.

On the subject of SPR-2915, I should note that I am NOT using Spring Web Flow,

Interestingly, I am using iBATIS, as was tortuga who reported this problem in the thread referenced above. But I don't really think that's related, at least not directly.

There are several issues with your code.

First of all PropertyEditors are not threadsafe!, so don't inject those, create a new one inside the initBind method. Also I would suggest that you supply the property name you want to bind to.

Next you are abusing your formObject to also hold referenceData.

Code:
protected Object formBackingObject(HttpServletRequest request) throws ServletException {
  EditChickForm form = new EditChickForm();
  Chick chick = new Chick();
  chick.setMother(new Hen(2, "Clucky", "red"));
  form.setChick(chick);
  return form;
}

protected Map referenceData(HttpServletRequest request, Object command, Errors errors) {
  Map referenceData = new HashMap();
  List<Hen> hens = new ArrayList<Hen>();
  hens.add(new Hen(1, "Fluffy"));
  hens.add(new Hen(2, "Clucky")); //leaving out color
  hens.add(new Hen(3, "Cranky"));
  referenceData.put("hens", hens);
  return referenceData;	
}

protected void initBinder(HttpServletRequest request, ServletRequestDataBinder binder) throws Exception {		
  HenEditor henEditor = new HenEditor();
  binder.registerCustomEditor(Hen.class, "chick.mother", henEditor);
}
Your jsp has to be edited accordingly

Code:
<p>Mother:<br />
  <f:select path="chick.mother">
    <f:options items="${hens}" itemValue="id" itemLabel="name"  />
  </f:select>
  <f:errors path="chick.mother" cssClass="error" />
</p>

You have some really valid points. At the end you externalize the object comparisons (normally done by equals()) to the property editors (done by string representations) and so make it reusable and changeable. The comparison criteria of an object type might change in different use cases (changeable). Or the same comparison criteria might apply to different object types (reusable), e.g. all persistent objects can be compared by their ID - and you neither need to duplicate this equals() code nor introduce an abstract base class.

I suggest to just raise a Jira issue of type enhancement and see if your idea gets support.

small oops
Well I was trying to understand better the PropertyEditor/Threading issue and I discovered a mistake in my previous code. Fixing it did not solve the problem, but for the sake of completeness...

In my property editor I had the method: getAsText(Object o)

It should be just: getAsText()

...and if I'm getting this right, that's exactly why threading is an issue. In your getAsText() method you would usually get the object stored in the current instance of the property editor using the parent class PropertyEditorSupport's getValue() method and then convert that into a String to return. If some other thread has set the value everything gets messed up (technically speaking).

Regardless, even w/ the right getAsText method, and not using dependency injection for the property editor, getAsText still does not get called for each object in the dropdown list. If you are reading this and find it frustrating too, go to the JIRA linked to above and vote.
