When you use the flow:executor tag in this minimal form, you're configuring the Web Flow system defaults that are consistent with the other environments Web Flow can be embedded (Spring MVC and Struts) and have proven to be the options generally preferred by most users.

Here are those defaults:

- A "ContinuationFlowExecutionRepository" which takes FlowExecution snapshots after each request to support use of browser navigational buttons (to allow for going back, restoring the snapshot at that previous point in time). You can read more about the various repositories in Chapter 4 of the reference manual. This repository does snapshotting by compressed serialization, which also requires objects in flow scope to be Serializable. A snapshot is a "continuation" btw (the terms are basically synonyms in this case).
- "AlwaysRedirectOnPause=true" which triggers a FlowExecutionRedirect after every event (request) is processed. This is basically automatic POST + REDIRECT + GET.

The combination of a continuation + alwaysRedirectOnPause gets you the nice back-button behavior with no browser warnings.

As far as what is going on here, it seems like for some reason when a continuation repository is used in a JSF environment the continuation identifiers are getting invalidated somehow. Its quite confusing to me how this happens so early, especially in James case, where it is happening immediatly. Clearly there is something wrong. The default max snapshot limit per conversation is 30.

As an exercise, it would be helpful if James or Msauer could try this executor configuration and see if the problem still exists:

Code:
<flow:executor id="flowExecutor" registry-ref="flowRegistry">
    <flow:repository type="continuation" max-continuations="-1"/>
</flow:executor>
This will place no limit on the max number of snapshots that can taken per conversation. I just want to make sure that's not related to the problem.

When you do not used the executor bean, the JSF configuration defaults to what it was prior to 1.0.2 for backwards compatability reasons: this is a "simple" repository (no snapshotting) with alwaysRedirectOnPause turned off. This is why Serializable was not an issue before, though it still is recommended objects in a conversational scope be Serializable because they still ultimately reside in the HTTP session typically.

In general, I would recommend always using the <flow:executor> tag and never relying on those "magic" JSF defaults. As Jeremy mentioned, the flow:executor can be used to configure basically any option of the Web Flow engine, including alwaysRedirectOnPause and the repository type used (for example, "continuation" [the default] or "simple"). See the webflow-config XSD and Chapter 5 of the reference manual for more information on the options.

As far as the variable resolver issue, it is expected most users will find DelegatingFlowVariableResolver enough for their needs, which is why it is now the recommended default and demonstrated in the samples. The other resolvers can be plugged in along side, of course, but the expectation is that's not what most JSF developers will want to do. In 1.1 we will add support for lazily-creating beans in any Web Flow conversational scope using Spring 2.0's custom scope extension point. The thought is DelegatingFlowVarableResolver for accessing flow variables combined with Spring as the managed bean provider (via DelegatingVariableResolver) will be the most common configuration, so 1.0.2 is paving the way for this while still providing full backwards compatability. I agree to the point that our JSF samples should be beefed up and will be for 1.1.

I will attempt to reproduce this strange 'continuation not found' problem. Any additional insight into this problem is appreciated.

I am able to confirm there is a bug related to this in the 1.0.2 JSF integration. I have fixed it and made a build available for testing. Hopefully, this resolves the ContinuationNotFoundException both James and Msauer are seeing.

Basically, a new flow execution key was being incorrectly generated on each flow execution "refresh" operation (triggerable by a flow execution redirect, a browser refresh button click, and possibly even a partial page refresh if the FlowPhaseListener was involved and the _flowExecutionKey is provided). With a continuation-based repository (the default), this would lead to a new continuation snapshot being taken and added. This would cause the ContinuationNotFoundException above once the number of refresh operations exceeded the maximum number of continuations allowed per conversation (30).

I was able to reproduce this easily by setting max-continuations to "1" and doing a single browser refresh click immediately after launching a flow.

The fix means the same key is now used throughout any number of refreshes of the same view, which is the correct behavior. A new key is only assigned after processing a navigation event, which allows a new snapshot to be taken to support going back to the previous view. This revised logic in FlowPhaseListener is consistent with the logic in FlowExecutorImpl, which is used by the other environments Spring Web Flow runs in.

Build 55 has this fix, available here: http://static.springframework.org/do...ng-webflow.php.

Can you guys try out Build 55 and let me know if this problem is resolved?

If you'd like to use the "flowScope" prefix, you need to configure FlowVariableResolver and FlowPropertyResolver in faces-config.xml. Alternatively, you can use the new FlowExecutionVariableResolver and FlowExecutionPropertyResolver and can then say: "flowExecution.flowScope.foo" or just "flowExecution.foo". The new FlowExecutionVariableResolver/PropertyResolver gives you explict access to more than just "flowScope" - you get access to flash and conversation scope, too. See the Java API documentation for more information on them.

You can use either of these other resolves with DelegatingFlowVariableResolver, which gives you flow execution attributes in #{foo.bar} style with no prefix. Or you can just use DelegatingFlowVariableResolver by itself. The believe is this is what most JSF developers would find most natural, as you don't tie your JSF views to what scope managed beans are located in. That's why sellitem-jsf only uses it now.

There's no bug here as far as I can see. It's just a choice about how you'd like to reference your flow variables (with an explict prefix or not), and then based on that you need to have the proper resolvers configured. Am I missing something?

James,

That wording isn't the best. The prefix is not optional with the DelegatingFlowVariableResolver. There is just no concept of a prefix period. This resolver attempts to resolve variables in the style #{someBean.someProperty} by searching flash, flow, and conversation scope for 'someBean'. If 'someBean' is not found this resolver simply delegates to the next resolver in the chain.

If you want an explicit prefix, you need to use one of the other resolvers as I mentioned. You can also combine usage of DelegatingFlowVariableResolver with one of these other resolver pairs to be able to do flow execution attribute resolution both with and without a prefix.
