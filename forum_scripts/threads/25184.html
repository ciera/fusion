I'm a little confused. Are you saying a flow scoped form object is being created more than once within a flow execution? If the ScopeType is Flow for the formObjectScope only one instance of the object should exist per flow execution. This is a pretty central feature, demonstrated by most of the samples like Phonebook, for example. The "invokeValidatorMethod" is passed the instance returned by the getFormObject method of FormAction, which will return a cached object if one exists in flow scope. There should be no issue there.

Can you summarize the issue that you are seeing clearly and also provide information about the version of SWF you are using.

But that's not what the framework does. The framework in FormAction.bindAndValidate will pull an existing formObject out of the formObjectScope if one is already there under the formObjectName. It will only create a new form object if one does not exist in the formObjectScope with that name. The test case above verifies this.

Also, the framework will bind to properties managed by a superclass if those properties are present as input parameters in the requestParameterMap.

With that said your using flow execution snapshots (continuations) to support back button use, so your flow
scoped form object is actually serialized out with the rest of flowScope when a snapshot is taken on "pause". On "resume" in the next request the execution is deserialized, which of course means any flow scope'd object instances are also deserialized. But they are still the same *logical* instances, with all state marked serializable preserved.

Could this serialization behavior perhaps be causing your problem? Again I simply can't reproduce your issue. In my tests calling setupForm binds a form object to flow scope. On bindAndValidate after a resume that formObject is retrieved from flow scope, as expected--within the same request it's the same exact instance, accross requests it's the same logical instance with all serializable state restored.
