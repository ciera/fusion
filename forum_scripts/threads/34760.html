Code:
<spring:bind path="command.fieldname">
   <c:out value="${status.error}"/>
</spring:bind>
status.error is the error object. You can also call some methods on it. Next you can also call some methods on the status object to retrieve all field, global, all errors, error count. (The status object is an instance of the [url=http://www.springframework.org/docs/api/org/springframework/validation/BindException.html]BindException [/urkl]class, so the methods mentioned on that class can be called on the status object).

Code:
<spring:bind path="command.*">
  ${status.errorCount}
</spring:bind>

I see the error in my ways...
So it seems that my problems yesterday all boiled down to two structures both writing to the status bean: the outer <c:for-each> loop and the inner <spring:bind> tag...

The following code concept gives me what I need (boiled it down to lose some of the extraneous junk - hopefully there are no typos...):

Code:
<form:form method="POST" commandName="timesheet">
  <c:forEach items="${timesheet.entry}" varStatus="loopStatus">
    <div>
      <form:input path="entry[${loopStatus.index}].hours">
    </div>
    <spring:bind path="timesheet.entry[${loopStatus.index}].hours">
      <c:if test="${status.error}">
        <c:set var="error">
          <c:forEach items="${status.errorMessages}" var="errmsg">
            <c:out value="${errmsg}"/>
          </c:forEach>
        </c:set>
      </c:if>
    </spring:bind>
    <div>
      ... use raw error data as necessary...
    </div>
  </c:forEach>
</form:form>
BTW, my purpose for using the raw error message(s) is that since I am working a spreadsheet-like form, I want to display any error messages within the cells' hover text. I am using the Overlib package to do this... See http://www.bosrup.com/web/overlib/ for more information...

Still have to ask... Is there any thought of adding a way to map the status bean from the <spring:bind> tag to another name (in the way that the JSTL <c:forEach> tag can using the varStatus attribute)? This would be handy, if only for code clarity purposes...

Hope this helps others in this situation...
